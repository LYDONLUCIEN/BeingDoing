# L3 / L4 开发启动计划（test_agent）

> 目标：在已完成的 Savepoint + L2 执行基础上，进入 L3（BDD）与 L4（AI 测试用户）正式开发。

---

## 1. 当前基线（已具备）

- Savepoint 闭环：create / load / export / delete
- L2 执行闭环：replay 与 playwright 双引擎
- generated 场景索引与管理页执行能力
- 批量任务：异步执行、取消、进度、历史、重启中断追踪
- 执行证据：report、trace、截图、stdout/stderr tail、完整日志落盘

这意味着：**L3/L4 可以直接复用现有数据入口、执行入口、证据出口**，无需从零搭建。

---

## 2. L3 / L4 尚未实现（本阶段目标）

### L3（BDD Runtime）

- Gherkin 解析器（Feature / Scenario / Given-When-Then）
- YAML 场景编排与 Gherkin 双轨输入
- Step Definitions 映射（业务语句 -> 标准动作）
- 断言契约层（业务断言 + 通用断言）
- L3 统一执行入口与结构化步级输出

### L4（AI Test User）

- TaskSpec 结构（persona / goal / golden_qa / budget / scoring）
- Agent Loop（Observe -> Plan -> Act -> Judge）
- LLM/CLI Bridge（当前建议先接 kimi CLI）
- L4 风险评分与失败分类
- L4 报告产物（timeline + artifacts + score）

## 4. 开发分期（建议按顺序）

## Phase 1：L3 最小可跑（优先）

目标：让 1 条 Gherkin 场景可执行、可断言、可出报告。

### 任务

1. 建立 `core/contracts` 与 `result_model`
2. 实现 `parser_gherkin.py`（最小语法集）
3. 实现 `adapters/beingdoing/step_definitions.py`（先覆盖 5~8 个高频步骤）
4. 复用 L2 runner 执行动作，输出步级结果
5. 新增 `run_l3.py` 入口

### 验收

- 命令可跑：`python test_agent/pipelines/run_l3.py --scenario <xxx.feature>`
- 输出报告包含：step status / duration / artifacts
- 首批场景覆盖：
  - 引导语触发
  - 表格失焦引导
  - 不重复提问

---

## Phase 2：L3 工程化增强

目标：L3 可用于持续回归。

### 任务

1. YAML 场景输入支持（与 Gherkin 并存）
2. AssertionContract 分层（通用断言 + 业务断言）
3. 失败分类标准化（selector / assertion / timeout / data）
4. 报告并轨到现有 generated 管理视图（可选）

### 验收

- 同一业务场景可用 Gherkin 与 YAML 两种表示执行
- nightly 可跑一组 L3 core 场景

---

## Phase 3：L4 最小可跑（先风险信号）

目标：L4 首版以“风险发现”为主，不做阻断门禁。

### 任务

1. 定义 TaskSpec（YAML）
2. 实现 Agent Loop（固定 max_turns / max_steps）
3. 接入 kimi CLI bridge（先 JSON 执行通道）
4. Judge 先规则判定（completion/consistency/ux_risk）
5. 产出 L4 报告（score + timeline + artifacts）

### 验收

- 至少 2 条 persona 样例可运行
- 可稳定输出报告与证据
- 失败时可定位（日志 + 行为轨迹）

---

## Phase 4：L4 稳定化

目标：让 L4 从实验性工具变成可持续资产。

### 任务

1. 重试与 fallback 策略
2. 失败聚类（TopN）
3. flakiness 基线统计
4. 与 L3/L2 的覆盖边界归档

## 6. 默认策略（先定死，减少争议）

- L3 主格式：Gherkin + YAML 双轨并存
- L4 初期定位：风险信号，不阻塞 PR
- L4 执行预算默认：
  - `max_turns=12`
  - `max_steps=40`
  - `timeout_sec=900`
- L4 重试默认：1 次（JSON 解析失败/瞬时执行失败）

---

## 7. 风险与注意事项

1. 不要在 L3/L4 core 里直接 import BeingDoing 业务模块
2. 所有业务语义都放 adapter 层（selectors / steps / assertions）
3. L4 判定以规则为主，LLM 打分只作解释增强
4. 先把 3~5 个高价值场景跑稳，再扩展用例规模

---

## 8. 里程碑定义

- M1：L3 单场景可跑（Feature -> 执行 -> 报告）
- M2：L3 core 场景 nightly 可跑
- M3：L4 2 条 persona 样例可跑
- M4：L4 风险报告稳定输出并可复现
