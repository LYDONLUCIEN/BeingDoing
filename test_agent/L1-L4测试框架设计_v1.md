# BeingDoing 自动化测试框架（L1-L4）设计 v1

## 1. 目标与边界

目标：构建四层测试体系，覆盖从稳定回归到智能模拟用户，且 L3/L4 尽可能通用、与业务代码解耦。

- L1：基础自动化（单元/集成/API 回归），高稳定、低成本，PR 门禁主力
- L2：浏览器自动化（页面行为与交互逻辑），验证真实 UI 流程
- L3：BDD（自然语言定义流程），每一步有结构化输出，作为可读的测试说明和执行日志
- L4：AI 用户模拟（按给定 persona/标准问答/目标行为），自动访问页面并执行完整探索式测试

约束原则：

1. **核心引擎通用化**：L3/L4 不直接依赖 BeingDoing 业务实现，使用插件/适配器接入
2. **业务适配外置化**：项目特定选择器、接口、断言、数据准备放在 adapter 层
3. **分层门禁**：L1/L2 进 CI 主链路；L3/L4 以 nightly/手动/预发为主
4. **证据优先**：所有层失败都要产出可读证据（日志、截图、trace、步骤输出）

---

## 2. 分层架构（推荐）

### L1 - Deterministic Regression（确定性回归）

职责：

- Python `pytest` 单元/集成/API 回归
- 复用现有 `test/backend` 与 `test/claude-test/backend`
- 覆盖状态机、业务规则、数据结构、prompt 模板稳定性

执行策略：

- PR 必跑（全量或按变更范围）
- mock 优先，保证稳定性与速度

### L2 - UI Behavior Regression（页面行为回归）

职责：

- Playwright 驱动页面操作
- 断言 DOM 行为、交互时序、核心可视流程
- 对高风险交互（blur/focus/dialog/localStorage）做强约束

执行策略：

- 前端相关 PR 必跑 smoke + 关键场景
- merge main 跑核心集，nightly 跑扩展集

### L3 - BDD Scenario Runtime（通用 BDD 运行层）

职责：

- 支持 Gherkin 或 YAML 的自然语言/半自然语言步骤定义
- 将步骤编译为标准 Action（goto/click/fill/assert/api_check）
- 每步输出：输入、执行结果、断言结果、耗时、证据路径

解耦点：

- L3 只认识 `ActionContract` 与 `AssertionContract`
- 不认识 BeingDoing 的具体页面结构和业务字段
- 业务差异由 adapter 提供 step definitions 与 selector map

### L4 - AI User Simulation（通用 AI 用户代理层）

职责：

- 以“测试用户”身份完成目标任务（goal-driven）
- 根据给定 persona、标准问答、限制条件，自动规划并执行页面行为
- 将每轮决策和行为转成可审计轨迹（Thought -> Action -> Observation）

解耦点：

- L4 由通用 Agent Runtime + Browser Driver + Adapter 工具组成
- 业务特定内容（阶段定义、术语、可操作对象）通过 adapter 注入

## 4. 核心接口（保证通用性的关键）

### 4.1 AdapterContract（业务接入标准）

- `prepare_data(mode, scenario) -> Context`
- `resolve_selector(alias) -> selector`
- `execute_domain_action(action, payload, page, context) -> StepResult`
- `assert_domain(assertion, context, artifacts) -> AssertResult`
- `fetch_runtime_state(context) -> dict`

只要实现这一组接口，L3/L4 就能迁移到其他项目。

### 4.2 标准动作 ActionContract（L3/L4 共享）

动作只保留通用语义：

- `goto`, `click`, `fill`, `press`, `wait_for`
- `assert_dom`, `assert_text`, `assert_network`
- `domain_action`（委托 adapter 执行业务动作）

### 4.3 标准报告 ResultModel

每步都记录：

- `step_id`, `input`, `status`, `duration_ms`
- `observation_summary`, `assertions`
- `artifacts`（screenshot/trace/network/log）
- `raw_llm_io`（L4 可选）

## 5. L3 设计（BDD）

建议两种输入：

1. **Gherkin**（便于产品/测试沟通）
2. **YAML 场景**（便于工程参数化）

执行流程：

1. 解析场景 -> 生成 step graph
2. step graph -> 标准 actions
3. actions -> driver + adapter 执行
4. 每步产出结构化日志与证据
5. 汇总报告并打分（通过率/关键断言）

示例（简化）：

```gherkin
Feature: Rumination 行为验证
  Scenario: 编辑假设后触发引导
    Given 我已进入 rumination 页面
    When 我编辑第 1 行假设为 "我想做独立咨询"
    And 我点击表格外空白触发失焦
    Then 我应看到右侧引导 "是否填写好了"
```

## 6. L4 设计（AI 测试用户）

L4 由两部分组成：

1. **Task Spec**：你给“标准问答答案/用户设定/提示词”
2. **Agent Loop**：读取页面状态 -> 决策下一步 -> 执行动作 -> 验证目标收敛

建议策略：

- `Planner`：根据目标拆成阶段任务（如登录 -> values -> rumination）
- `Actor`：将任务转动作（借助 adapter 的 domain_action）
- `Judge`：判断是否达成目标、是否偏航、是否触发异常

稳定性控制：

- 限制最大步数、最大轮次、每轮 token
- 失败兜底：回退到脚本化动作
- 关键断言仍由 deterministic 规则决定，避免“纯 LLM 判题”

## 7. 与现有仓库的对接策略

现状优势（可直接复用）：

- 已有强 L1：`test/backend` + `test/claude-test/backend`
- 已有 replay/savepoint/fixture 资产
- 已有一份 L2/L3 规划文档（可转为 adapter 场景库）

建议对接：

1. `l1/run_l1.py` 先统一编排现有 pytest 与报告
2. `adapters/beingdoing/data_seed.py` 复用现有 fixture/fork
3. L2 先做 3-5 条 P0 场景（最痛点交互）
4. L3 先把这 3-5 条改写成 BDD 场景，验证“同一场景双表示”
5. L4 先做 nightly 小样本（2-3 条），建立稳定基线后再扩

## 8. CI/CD 分层门禁

- PR: `L1 full + L2 smoke`
- main: `L1 full + L2 core`
- nightly: `L2 extended + L3 core + L4 sample`
- release-candidate: `L1/L2/L3 + L4 focused`

注意：

- L3/L4 默认不阻塞 PR 合并，只出风险报告（初期）
- 达到稳定阈值后可逐步提升为软门禁

## 11. L3 具体案例（非技术可编写）

目标：让产品/运营/测试同学不用写代码，也能写出可执行测试流程。

### 案例 A：价值观流程（业务语言）

```gherkin
Feature: 价值观探索流程
  作为测试同学
  我希望验证“重复追问”不会出现
  以便确保用户体验流畅

  Scenario: 用户给出模糊回答后，系统应追问但不重复同题
    Given 我使用测试账号 "qa_values_01" 登录系统
    And 我进入 "价值观探索" 页面
    When 我输入 "我更喜欢有影响力的工作"
    And 我点击发送
    Then 我应该看到助手继续追问细节
    And 在后续 3 轮内，不应出现“完全同一句问题重复两次”
    And 当前阶段应保持在 "values" 或进入下一步
```

这个场景里的句子都用“业务语义”，不出现 selector、CSS、API 名称。  
技术细节由 `adapters/beingdoing/step_definitions` 负责映射。

### 案例 B：Rumination 表格失焦引导（业务语言）

```gherkin
Feature: Rumination 引导顺序
  Scenario: 编辑假设后失焦，应出现正确引导
    Given 我已进入 "rumination" 阶段
    And 当前有可编辑的假设表格
    When 我把第 1 行假设修改为 "我想做独立咨询"
    And 我点击表格外空白区域
    Then 右侧应出现引导语 "是否填写好了"
    When 我回复 "确认，可以继续"
    Then 系统应推进到下一步骤
```

### 案例 C：BDD 执行后的每步输出（示意）

```text
[STEP 01] Given 我已进入 "rumination" 阶段
- action: goto("/explore/chat/rumination")
- result: PASS (2.1s)
- evidence: screenshots/step01.png

[STEP 02] When 我把第 1 行假设修改为 "我想做独立咨询"
- action: domain_action("table_edit", {"row":1,"field":"hypothesis","value":"我想做独立咨询"})
- result: PASS (1.4s)

[STEP 03] Then 右侧应出现引导语 "是否填写好了"
- action: assert_text("是否填写好了")
- result: PASS (0.3s)
```

## 12. L4 具体案例（单代理：Test User Agent）

L4 采用单代理模式：

- **Test User Agent**：模拟真实用户，自动浏览页面、输入内容、完成任务、收集反馈

被测对象可以是：

1. 含 AI 引导能力的一体化网页应用
2. 传统表单/流程型网页
3. 混合交互系统（部分步骤是规则逻辑，部分步骤由 LLM 生成）

L4 不要求被测系统内部必须存在“Product Agent”。  
L4 的职责是从用户视角验证：引导是否清晰、反馈是否正确、流程是否可完成、体验是否一致。

### 12.1 通用任务规范（TaskSpec）示例

```yaml
task_id: l4_values_persona_001
goal: 在 12 轮内完成价值观阶段，并生成可解释的阶段结论
persona:
  age: 28
  role: 产品经理
  style: 表达简洁，偶尔犹豫
  constraints:
    - 不主动提供与问题无关的个人隐私
    - 回答长度控制在 20~80 中文字
golden_qa:
  expected_signals:
    - 能给出至少 2 个价值偏好
    - 系统有至少 1 次追问动机
  forbidden_signals:
    - 连续两轮完全相同提问
    - 泄露状态调试标签
budget:
  max_steps: 40
  max_turns: 12
  max_runtime_sec: 600
  llm_max_tokens_per_turn: 200
scoring:
  completion_weight: 0.4
  consistency_weight: 0.3
  ux_risk_weight: 0.3
stop_conditions:
  - reached_phase: talents
  - hard_error: true
```

### 12.2 L4 Agent Loop（通用）

1. 读取页面状态（当前阶段、最近对话、可交互元素）
2. 根据 `goal + persona + golden_qa` 生成下一步意图
3. 选择动作（输入文本、点击、等待、导航）
4. 执行动作并观察结果
5. Judge 打分：是否逼近目标、是否触发 forbidden 信号、是否存在可用性风险
6. 达成目标/触发停止条件则结束并输出报告

### 12.3 输出报告（示意）

```json
{
  "task_id": "l4_values_persona_001",
  "status": "passed_with_risk",
  "score": 0.82,
  "signals": {
    "completion": true,
    "consistency": true,
    "ux_risk": ["第7轮响应较慢(11.2s)"]
  },
  "timeline": [
    {"turn": 1, "action": "chat_send", "user_text": "我在意工作的社会价值"},
    {"turn": 2, "action": "chat_send", "user_text": "我希望帮助真实的人"}
  ],
  "artifacts": {
    "trace": "artifacts/l4_values_persona_001/trace.zip",
    "screenshots": "artifacts/l4_values_persona_001/screenshots/",
    "conversation_dump": "artifacts/l4_values_persona_001/conversation.json"
  }
}
```

## 13. “通用框架”如何做到跨项目复用

跨项目复用时，只替换 `adapter` 层：

1. `selectors.yaml`：元素别名映射
2. `domain_actions.py`：项目专属动作（如表格编辑）
3. `assertions.py`：项目专属断言（如 state_code、metadata）
4. `task_library/`：行业/产品的 persona 与目标模板

`core/bdd` 与 `core/ai_user` 不改动，做到一次建设、多项目复用。

## 13.1 L4 适用边界（与 L3 的分工）

- L3 负责“已知流程”的可重复验证（规则清晰、断言明确）
- L4 负责“未知分支/体验质量”的探索式验证（是否易懂、是否卡住、是否有下一步引导）

建议：

- 对关键规则（权限、状态、本地持久化）优先用 L2/L3 兜底
- 对体验连贯性、引导自然度、多分支健壮性，用 L4 扩展覆盖

## 15. 实施规范补充（针对专家问询）

### 15.1 L2 规范化落地

- **组织形式**：`YAML 场景 + Python helper`
- **执行入口**：L2 必须经过 `orchestrator + ActionContract`
- **分级标准**：
  - `smoke`：1~3 条，阻断级，<3 分钟
  - `core`：P0/P1 主路径，<15 分钟
  - `extended`：nightly 扩展集

### 15.2 L1 报告并轨

- `l1/run_l1.py` 调用 pytest 并消费 JSON/JUnit 输出
- 通过 `pytest_adapter` 转换到统一 `ResultModel`
- 总报告统一由 `pipelines/run_all_levels.py` 生成
- 原生 pytest 报告作为附件保留

### 15.3 core 与 adapter 边界

- `core` 负责：浏览器驱动、动作执行、重试/超时、统一编排
- `adapter` 负责：环境注入（URL/auth）、业务动作、业务断言、seed/fork
- 严格禁止 core 直接依赖业务模块

### 15.4 L4 执行与可靠性

- 仅单代理 `Test User Agent`
- 允许通过外部 CLI Agent（如 `kimi`）接入
- Bridge 执行协议：**只认 JSON 执行通道**
- 失败策略：JSON 校验失败重试 1 次，之后 fallback
- fallback：场景级优先，全局兜底

### 15.5 BDD 映射机制

- 匹配优先级：精确模板 > 正则 > 同义词词典 > 失败报错
- 一个 step definition 可映射多个 actions
- 代码位置：`adapters/beingdoing/step_definitions.py`（或子目录）

### 15.6 数据隔离

- 每次执行生成 `run_id`
- 数据与 artifacts 使用 `run_id/case_id` 隔离
- L4 探索数据默认临时目录，失败样本可配置保留

### 15.7 证据规范（强制）

- 关键步骤截图（必须）
- 失败前后截图（必须）
- 结束态截图（必须）
- Playwright trace（默认开启，失败必保留）

### 15.8 配置覆盖模型

- `base.yaml`：全局默认
- `local.yaml` / `ci.yaml`：环境覆盖
- `scenario.yaml`：场景覆盖
- 优先级：`scenario > env > base`

### 15.9 L4 评分落地

- `completion`：规则判定
- `consistency`：规则判定（重复、矛盾、非法状态）
- `ux_risk`：规则触发（超时、卡死、无引导等）
- LLM 评分仅作解释增强，不作为硬门禁真值

### 15.10 失败闭环与趋势

- 自动失败分类：`network_timeout` / `assertion_failed` / `selector_not_found` / `llm_invalid_json` 等
- nightly 趋势：通过率、耗时、失败类型 Top N、flaky 清单

## 16. 场景归属决策树（主责任层 + 补充层）

为避免“这个场景到底是 L2/L3/L4”的争议，采用以下规则：

1. 每个场景必须指定一个 `primary_level`（主责任层）
2. 可选一个或多个 `support_levels`（补充层）
3. 通过“测试意图”而非“测试工具”决定层级

### 16.1 三级判定（先问意图）

- **Q1：核心是验证 UI 事件是否触发正确状态吗？**  
  是 -> `L2` 主责任
- **Q2：核心是验证流程契约/状态迁移是否正确吗？**  
  是 -> `L3` 主责任
- **Q3：核心是探索未知分支与体验鲁棒性吗？**  
  是 -> `L4` 主责任

若同一场景同时命中多项，按“失败代价最高的一项”设为 `primary_level`，其余作为 `support_levels`。

### 16.2 用户当前高关注场景映射

#### 场景 A：每个 phase 的引导语是否正常

- `primary_level: L3`
- `support_levels: [L2]`
- 原因：主目标是“引导契约一致性”，而不是单个按钮行为

#### 场景 B：rumination step3 各种操作都能触发引导

- `primary_level: L2`
- `support_levels: [L3]`
- 原因：先验证操作触发链路，再验证触发后文案/步骤契约

#### 场景 C：边界行为（结论卡反复拒绝、过滤后 0/1~3 条跳转）

- `primary_level: L3`
- `support_levels: [L4]`
- 原因：状态迁移规则可确定性断言；L4 用于补充未知组合路径

### 16.3 场景元数据（建议必填）

```yaml
id: rumination_filter_edge_001
primary_level: L3
support_levels: [L4]
intent: 状态迁移契约
risk: P0
start_from: savepoint:5-3-e
```

## 17. 管理员页面 Savepoint（与 Fork 一体化）

目标：让不懂代码的管理员在调试页面直接生成检查点，并可立即 Load 回放；同一检查点同时用于 L2/L3/L4 自动化。

### 17.1 权限与可见性（硬约束）

- 仅 `super_admin` 可见按钮
- 仅在调试工作区（`SBX` Fork 或 `ADM` 常驻工作区）可用
- 普通用户与正式激活码页面完全不可见

### 17.2 核心语义

在某条 AI 消息下点击 “保存为检查点（Savepoint）” 时：

1. 以该消息为参考断言目标（可保存摘要/关键词）
2. 保存状态为“该消息出现之前”的状态
3. 若该消息前一条是用户输入，默认连该用户输入一起回退（便于重新输入复现）

即：Load 后重新输入同一用户消息，应触发同一路径并产出接近参考输出。

### 17.3 两类动作

- `Save`：生成一个检查点资产（持久化）
- `Load`：将当前线程回退到该检查点（即时调试）

Load 生效模式固定为：

- 替换整个 report 状态（含后续 phase 清理结果）
- 自动跳转到保存时的 `phase`
- 自动定位到保存时的 `thread_id`

### 17.4 建议后端接口（草案）

- `POST /admin/savepoints/create`
  - 入参：`activation_code, phase, thread_id, target_message_id, rewind_mode`
  - 出参：`savepoint_id, savepoint_name, artifact_paths, reference_assertion`

- `POST /admin/savepoints/load`
  - 入参：`activation_code, savepoint_id`
  - 出参：`loaded=true, phase, thread_id, message_count_after_load`

- `GET /admin/savepoints`
  - 支持按 `phase/risk/tag` 列表与筛选

### 17.5 存储建议

- 管理员运行态：`data/test/simple/sandboxes/...`（现有）
- Savepoint 资产：`test/backend/fixtures/simple_chat_reports/admin_sp_*`
- 元数据索引：`test/backend/fixtures/simple_chat_cases/batch_savepoints_general.json`（可自动追加）

这样可直接复用现有 `generate_simple_chat_savepoints.py` / `replay_simple_chat.py`。

### 17.6 与自动化测试的连接

管理员点 Save 后，系统自动产出：

1. 标准 fixture 目录（可 seed）
2. 标准 case 片段（可回放）
3. 标准 scenario 引用（L2/L3/L4 可直接使用）

实现“管理员手动造检查点 -> 自动化直接消费”的闭环。

### 17.7 保证“与非管理逻辑一致”

Load 检查点后，后续对话必须继续走正式接口：

- 继续调用 `/simple-chat/message/stream`
- 不允许管理端注入“快捷回复”或跳过核心业务分支
- 管理端仅负责“数据回退/定位”，不改业务推理主流程

### 17.8 第一版范围控制（建议）

- 先支持 `values` 与 `rumination`
- 先支持单线程回退（不做跨线程合并）
- 先支持 “回退到目标 AI 消息前（可连带删除前一条用户消息）” 一种模式

稳定后再扩展多模式回退。

## 18. 术语统一：seed / case / scenario

### 18.1 seed（测试起跑数据）

- 含义：一份可复用的“起始状态数据快照”（通常是 `report` 目录）
- 作用：让测试从中间状态起跑，避免每次从第 1 轮重走
- 形态：`test/backend/fixtures/simple_chat_reports/<name>/...`

### 18.2 case（可执行回放条目）

- 含义：一条结构化执行指令（phase、thread、message、断言）
- 作用：用于 replay/pytest 的稳定回归
- 形态：`test/backend/fixtures/simple_chat_cases/*.json` 中的单条 case

### 18.3 scenario（测试场景编排）

- 含义：多步骤流程定义（可用于 L2/L3/L4）
- 作用：描述完整测试意图与执行步骤（包含 stop 条件、评分、证据策略）
- 形态：`adapters/beingdoing/scenarios/l2|l3|l4/*.yaml`

### 18.4 三者关系

- `seed` 提供“从哪里开始”
- `case` 提供“最小可执行回放”
- `scenario` 提供“完整流程编排”

管理员保存 Savepoint 时，建议自动产出这三类资产，确保手工调试成果可自动化复用。

## 19. Savepoint 回退语义（纯净性）

为满足“从某检查点向后长链路测试”的纯净需求，定义两种回退模式：

### 19.1 `global_rewind`（唯一模式）

- 以“选中消息所在轮次”为分界点
- 删除该轮及之后的所有对话消息
- 同时清理后续 phase / step 的进度与衍生状态（包括 conclusion/rumination 后续状态）
- 目标：确保从检查点继续测试时不受未来数据污染

### 19.2 当前约束

- 仅保留 `global_rewind`，不提供 `round_rewind`
- 不提供“最小回退”选项，避免管理员误用导致伪复现

## 20. 数据隔离与真实链路一致性（硬性要求）

### 20.1 数据隔离

- Fork / Savepoint / 回放数据仅落在测试根目录（如 `data/test/simple/...`）
- 禁止写入正式 `data/simple` 用户数据
- 管理端仅可在 `SBX/ADM` 工作区执行 Savepoint 操作

### 20.2 真实链路一致性

- 使用与用户相同的业务 API：`/simple-chat/message/stream`、history、threads
- 使用与用户相同的后端函数与路由，不允许管理端专用“捷径推理”
- 管理能力仅用于“准备/回退数据”，不替代正式推理执行链路

这条是框架有效性的前提：若执行链路不同，测试结论无效。

## 21. Savepoint 命名与索引管理（管理员可读）

### 21.1 管理员自定义命名（必填）

- 点击 Savepoint 时，管理员必须输入 `display_name`（中文可读）
- 命名用于表达测试目的，如：
  - `step3-失焦后引导语触发`
  - `结论卡-连续拒绝后再次确认`
- `display_name` 在索引中必须唯一（避免重复命名造成误用）

### 21.2 ID 与目录

- `savepoint_id` 由系统生成，作为后端目录名（机器稳定标识）
- `display_name` 由管理员输入，作为运营/测试可读标识
- 两者必须分离：重命名不影响目录与引用

### 21.2.1 断言参考生成（固定策略）

- 点击 Savepoint 时，系统自动从被选 AI 消息提取 `expected_hint/keywords`
- 管理员可在保存弹窗中编辑后确认
- 最终保存值写入：
  - `data/test/simple/savepoints/<savepoint_id>/meta.json`
  - `data/test/simple/savepoints/index.json`
  - `test_agent/savepoints_registry.json`

### 21.3 索引文件（登记册）

在 Savepoint 根目录维护索引 JSON（单一真值），固定为：

`data/test/simple/savepoints/index.json`

同时固定输出评审镜像（用于 PR 可读审查）：

`test_agent/savepoints_registry.json`

记录字段建议：

- `savepoint_id`
- `display_name`
- `description`（可选）
- `created_by` / `created_at`
- `source_activation_code`
- `source_fork_id`
- `phase` / `thread_id`
- `target_message_id`
- `rewind_mode`（固定为 `global_rewind`）
- `fixture_path`ju

















- `tags`（可选）

### 21.3.1 命名校验

- 创建时若 `display_name` 已存在，接口返回冲突错误（如 409）
- 不做自动后缀补全，命名冲突必须由管理员手工改名后重试
- 冲突响应需返回已存在条目的关键信息：`savepoint_id`、`created_at`、`display_name`

### 21.4 Fork 数据不被覆盖（硬约束）

- Savepoint 创建是“从 Fork 复制并裁剪后落盘到 Savepoint 目录”
- 不允许就地修改 Fork 原始数据
- Load 时可将 Savepoint 投影到当前测试工作区，但 Fork 作为源数据保持不变

### 21.5 双索引一致性

- `data/test/simple/savepoints/index.json`：运行态真值（系统读取）
- `test_agent/savepoints_registry.json`：评审镜像（人可读/代码评审）
- 每次创建/更新/删除 Savepoint 时，必须同步更新两者

### 21.6 管理页显示要求

- Savepoint 列表必须显示：
  - `display_name`
  - `savepoint_id`
  - `created_at`（创建时间，必显）
  - `phase` / `thread_id`（建议）

### 21.7 目录结构（固定）

固定目录布局如下：

- `data/test/simple/savepoints/<savepoint_id>/report/`
- `data/test/simple/savepoints/<savepoint_id>/meta.json`
- `data/test/simple/savepoints/index.json`

其中：

- `report/`：裁剪后的可加载数据（用于 load/replay/自动化）
- `meta.json`：该检查点单点元数据（名称、来源、创建信息、断言参考）
- `index.json`：全局索引（运行态真值）

## 22. 当前已实现功能（MVP）与使用方法

本节记录已经开发完成且可直接使用的能力。

### 22.1 管理员 Savepoint（页面能力）

**目的**：让管理员在页面内沉淀检查点，并快速复测。

**入口**：

- 聊天页 AI 消息工具栏：`保存为检查点`

- Delete：删除检查点（含删除影响确认）
- 列表：搜索、phase 筛选、分页、时间排序

### 22.2 回放命令与结果回写

**目的**：把检查点转成可重复命令，并自动沉淀回放结果。

**命令来源**：

- Savepoint 行级按钮：`复制回放`

**目的**：统一查看所有回放执行结果，快速定位失效检查点。

**批量场景**：
