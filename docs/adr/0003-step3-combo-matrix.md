# ADR-0003: Step3 组合矩阵与深度讨论拆分

**状态**: 已批准
**日期**: 2026-06-16
**决策者**: 产品 + 开发 Grill

## 背景

Step3（假设生成）原来是扁平表格逐行解锁模式：后端预生成 N 行组合，用户按 cursor 逐行填写假设。问题：
- 用户容易失焦，每行对话混杂在同一消息流中
- 无法自由切换组合查看

## 决策

将 step3 拆分为两个子步（`filter_step=3` 和 `filter_step=3b`），进度条统一显示在 3。

### filter_step=3：组合假设填写

- **矩阵选择器**：左侧上半部展示 3个热爱胶囊 + 5个优势胶囊，用户点选确定当前组合
- **默认遍历顺序**：热爱优先（热爱1×优势1, 热爱1×优势2, ..., 热爱1×优势5, 热爱2×优势1, ...）
- **独立对话**：每个组合有独立对话记录，AI 上下文完全隔离
- **结论卡**：textarea 可编辑，>=5字可确认；点击 chips 直接替换填入
- **三种状态**：empty / confirmed / skipped（有字跳过保留文字+灰显+"已跳过"标签）
- **弹窗**：首次确认/跳过后弹出完成提示，可勾选"不再提示"
- **全部提交**：始终可点，弹窗追问纯空和有字跳过的组合
- **引导语**：固定模板，替换完成数量 n

### filter_step=3b：深度讨论

- 等价于当前 step3 的表格+全局对话模式
- 表格只含已确认假设的组合（列：热爱、优势、假设），全部解锁
- 左上角返回按钮可回到 3（状态保持）
- neg gate 仅在此阶段触发
- 与 step4+ 衔接逻辑完全复用

## 技术方案

### 消息存储
- 一个消息文件（`main_flow.json`），所有组合消息共存
- 每条消息带 `combo_id`（`passion_idx + strength_idx` 拼接，如 `"02"`）
- 文件头部维护 combo_id → 热爱/优势名称映射

### combo_id 格式
- `passion_idx`（0-2）+ `strength_idx`（0-4），字符串拼接
- 示例：`"00"`, `"04"`, `"23"`

### AI 上下文隔离
- 后端按 `combo_id` 切片消息，只发送当前组合的对话历史 + 用户背景
- 不跨组合共享假设内容

## 后果

- 前端需要新建矩阵选择器组件和可编辑结论卡组件
- 后端消息存取层需支持 `combo_id` 字段
- 对话 API 需新增 `combo_id` 参数
- 进度存储结构需扩展（combo 状态映射）
- 3b 复用现有表格和 neg gate 逻辑，改动较小
