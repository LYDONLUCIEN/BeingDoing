# ID 统一收口实施计划（report_id / thread_id / activation_session_id）

## 1. 目标与范围

本计划聚焦“ID 语义去歧义”，避免 `session_id` 一词多义导致的维护与排障成本。

目标：

- 业务语义统一为三元组：`report_id`、`thread_id`、`activation_session_id`
- 兼容存量字段：短期保留旧 `session_id`，但业务层不再直接依赖其语义
- 通过统一读写入口减少散落兼容判断

范围：

- 后端：`simple_auth`、`simple_chat_routes`、公共工具层
- 前端：探索流程 session 读写与消息流请求参数
- 文档与测试：补充验收标准与回归用例

---

## 2. 实施策略（推荐：收口 + 渐进迁移）

采用“三阶段”：

1. **阶段 A：统一读写收口（先做）**
   - 新增统一 ID 工具类/函数
   - 业务代码改为调用统一入口（不再手写兼容判断）
2. **阶段 B：历史数据迁移（可灰度）**
   - 脚本回填新字段（保留旧字段）
   - 对抽样与全量数据进行一致性校验
3. **阶段 C：收尾下线旧字段（稳定后）**
   - 读取逻辑逐步不再回退旧字段
   - 文档与日志只保留新语义

为什么不直接全量迁移：

- 当前文件结构多样，历史数据版本不完全一致；
- 直接硬迁移风险高，且回滚成本高；
- 先收口可立即降低维护成本，再迁移更稳。

---

## 3. 阶段 A 详细改造步骤（本轮执行）

## A1. 后端新增统一 ID 静态工具类

新增文件（建议）：

- `src/backend/app/utils/id_codec.py`

包含能力：

- `read_activation_session_id(payload, fallback=None)`
  - 仅读取显式 `activation_session_id`（避免把消息里 legacy `session_id` 误当成 activation）
- `read_activation_session_id_from_activation_record(payload)`
  - 用于 `activations.json` / ActivationRecord：`session_id` 表示 activation 存储会话
- `read_thread_id(payload, fallback=None)`
  - 优先 `thread_id`，回退 `session_id`（仅在线程语义上下文使用）
- `build_message_ids(thread_id, activation_session_id, include_legacy=True)`
  - 统一生成写入消息日志所需 ID 字段，避免业务层重复拼接
- `build_activation_client_view(rec, logical_thread_id)`
  - 统一生成 simple-chat API 嵌套 `activation` 对象（保留 legacy `session_id`=thread，新增显式字段）
- `build_note_container_root(report_id)`
  - 新建 `*__note/note.json` 时写入 `report_id` + legacy 顶层 `session_id`（= report_id）

## A2. 后端接入收口函数

改动文件：

- `src/backend/app/api/v1/simple_chat_routes.py`
- `src/backend/app/api/v1/simple_auth.py`

改动方式：

- 读取前端请求 ID 时通过 `id_codec` 解析，不直接 `or` 链
- 写 message payload 时统一调用 `build_message_ids(...)`
- 日志打印统一输出三元组（至少包含 `thread_id` + `activation_session_id`）

## A3. 前端新增 session ID helper

改动文件（建议）：

- `src/frontend/lib/explore/session.ts`

新增能力：

- `getActivationSessionId(session)`：优先 `activationSessionId`，回退 `sessionId`
- `setActivationSessionId(session, id)`：统一双写（`activationSessionId` + 兼容 `sessionId`）

接入点：

- `src/frontend/app/(main)/explore/chat/[phase]/page.tsx`
  - 保存 session 时改用 `setActivationSessionId`
  - 请求 `/message/stream` 时通过 `getActivationSessionId` 传 `activation_session_id`

---

## 4. 阶段 B（迁移脚本）执行条件与方案

满足以下条件再做全量迁移：

- 阶段 A 上线并稳定运行 >= 1 个版本周期
- 错误日志中不再出现 ID 语义混淆问题
- 已有可回滚备份（`data/simple` 全量备份）

迁移脚本目标：

- 对 `data/simple/reports/*` 回填 `thread_id` / `activation_session_id`
- 保留原 `session_id` 字段，新增结构化字段
- 生成迁移报告（成功数、跳过数、异常文件列表）

---

## 5. 验收标准

代码层验收：

- 业务代码中不再新增“裸兼容判断” (`x or y or z`) 处理核心 ID
- 核心路径统一通过 helper/codec 读写 ID
- 新增字段在前后端请求与日志中前后对应

功能层验收：

- 老数据可正常读取（无回归）
- 新写入消息含 `thread_id` 与 `activation_session_id`
- 前端刷新/切线程后不会出现 ID 绑定错乱

可观测性验收：

- 关键日志可同时定位 `report_id/thread_id/activation_session_id`
- 出现问题时可单步还原链路

---

## 6. 测试计划

## 6.1 静态检查

- 前端：TS lint 无新增错误
- 后端：Python lint/类型检查无新增错误（忽略环境缺包型告警）

## 6.2 手工回归（必做）

1. 使用旧激活码进入已存在旅程，发送消息，确认对话正常；
2. 新建线程发送消息，检查后端日志 ID 三元组是否正确；
3. 触发结论卡生成与确认，检查 message / note 落盘字段；
4. 刷新页面后继续对话，确认不会绑定到错误线程；
5. 切换阶段后继续对话，确认 `activation_session_id` 与 `thread_id` 未串线。

## 6.3 数据文件抽样检查

- 抽样 `record.json`、`phase__thread.json`、`__note/note.json`
- 验证：
  - `thread_id` 与 category 后缀一致
  - `activation_session_id` 与激活记录一致
  - 旧 `session_id` 仍存在（兼容阶段）

---

## 7. 风险与回滚

风险：

- 老代码仍读旧字段，若收口不全可能出现局部不一致
- 迁移前后混写期可能出现字段重复来源

回滚策略：

- 阶段 A 改动仅新增字段与收口函数，不删除旧字段，可快速回退代码版本
- 阶段 B 迁移前做 `data/simple` 全量备份；异常时回滚文件快照
