**为什么最关键**:T2 改了 auth_service.py 的 token 时间判断,改不好所有用户登录失败。代码层分析安全,但没跑过。

### 方式二:curl 验证(更精确)
```bash
# 登录拿 token
curl -X POST http://localhost:8000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"你的邮箱","password":"你的密码"}'

# 用 token 调需 auth 接口(等几秒后调,验证 exp 判断)

  http://localhost:8000/api/v1/sessions
```

**预期**:返回 200 + 数据,不是 401。

**如果 401 或报错**:
- 看后端日志:`tmux attach-session -t beingdoing` 或 `journalctl`

## B5. 检查/配置生产环境变量 🔴

T4 邮件功能依赖 SMTP,新前端依赖 Server Action key。确认 `/etc/beingdoing.env`:

```bash
grep -E "SMTP_|NEXT_SERVER_ACTIONS" /etc/beingdoing.env
```

**应该有这些**(没有就补上):

```bash
# T4 邮件功能必需
SMTP_HOST=smtp.163.com
SMTP_PORT=465
SMTP_USER=soulhappylab@163.com
SMTP_PASS=<163邮箱授权码>
SMTP_FROM_EMAIL=soulhappylab@163.com
SMTP_FROM_NAME=寻录
SMTP_USE_SSL=True

# 前端 Server Action 必需(没设会前端报错)
NEXT_SERVER_ACTIONS_ENCRYPTION_KEY=<openssl rand -base64 32 的输出>
```

**为什么需要 `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY`**:CLAUDE.md 明确写,生产模式重新 build 后,如果没设这个 key,Server Action 会解密失败,前端报 `Failed to find Server Action`。本次 T1/T3/T4 新增了多个 Server Action,更不能漏。

# 一句话流程速查

## Part A(本地验证)
```
重跑pytest → start-dev → 验证JWT登录 → 浏览器验5个功能 → push origin main
```

## Part B(生产部署)
```
ssh → 停服务 → 备份DB → git pull → 查.env → alembic upgrade → 清.next重build → 启动 → 看日志 → 浏览器验 → (出错回滚) → 打tag
```

## 3 个绝对不能跳的步骤(生产)
1. **B3 备份 DB** — migration 005 不可逆,出问题靠备份救命
2. **B7 清 `.next` 重 build** — 否则前端 Server Action 报错
3. **B9 看后端日志** — T2 时区改造可能炸 JWT,第一时间发现

## 3 个必须提前准备好的东西(生产)
1. **B5 的 `NEXT_SERVER_ACTIONS_ENCRYPTION_KEY`** — 没设前端报错
2. **B5 的 SMTP 配置** — T4 邮件要用(163 邮箱授权码)
3. **B11 回滚命令** — 出错时直接复制粘贴
