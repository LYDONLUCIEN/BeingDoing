## 一、提示词问题大类（重点）

### P-02 使命阶段：要求“只能选一个价值观”，与期望可多选不一致
- 问题分析：
  - 当前使命阶段系统提示词在“匹配价值观”处强调“某个价值观”，虽然文本中有“某个（或哪些）”痕迹，但执行上容易收敛成单选。
  - 需求已拍板：不新增结构化多值字段，只允许用户在文案里自由表达多个价值观。
- 修改计划：
  - 提示词层：明确“可写多个价值观”，且不要求主次结构。
  - 数据层：保持现有字段结构不变，不做多选结构化改造。
  - 回归：覆盖使命阶段对话输出、rumination 第4步文案一致性、最终报告展示一致性。
- 调整风险：
  - 低风险：主要为提示词与文案约束调整。
  - 残余风险：由于不结构化存储，多值统计与检索精度会低于结构化方案。
- 可能修改位置（代码）：
  - `src/backend/app/domain/prompts/templates/simple_chat_system.yaml`
  - `src/backend/app/domain/rumination_prompt_strings.py`
  - `src/backend/app/utils/rumination_table_widgets.py`
  - `src/frontend/components/explore/RuminationTableWidget.tsx`
- 可能修改位置（提示词）：
  - `uidesign/prompt/0412-purpose.md`
  - `uidesign/prompt/new-rumination2.md`

### P-03 使命阶段：价值关键词统计展示要移除
- 问题分析：
  - 当前使命提示词仍包含“统计每个价值观关键词出现次数”的流程描述。
  - 产品诉求是“不展示统计内容”，需避免模型输出统计语言或频次。
- 修改计划：
  - 删除使命提示词中的“对用户展示统计”动作，改为“归纳 + 用户确认”。
  - 保留内部统计供模型推理使用，但禁止输出到用户可见文案。
- 调整风险：
  - 低风险（提示词变更为主）。
  - 注意不要影响使命阶段收敛质量。
- 可能修改位置（代码）：
  - `src/backend/app/domain/prompts/templates/simple_chat_system.yaml`
  - `src/backend/app/api/v1/simple_chat/prompt_builder.py`
- 可能修改位置（提示词）：
  - `uidesign/prompt/0412-purpose.md`

### P-04 Rumination 否定弹窗与“深入聊聊”文案格式不一致
- 问题分析：
  - 当前否定闸门文案是“整行 line summary”拼接，和期望“把假设单独拿出来确认”不一致。
  - `new-rumination-3.md`对步骤2/3/5/6均定义了更精细的话术模板。
- 修改计划：
  - 将闸门文案生成从“line摘要”改为“字段级模板”（按步骤分别生成）。
  - 步骤3优先展示“否定假设文本 + 对应热爱/优势”，避免整行噪音字段。
  - “深入聊聊”开场语统一采用参考文档口径，并按每条循环一次一问。
- 调整风险：
  - 中风险：影响已上线闸门交互，需回归“continue/deep_start/deep_end”三个分支。
- 可能修改位置（代码）：
  - `src/backend/app/utils/rumination_neg_gate.py`
  - `src/frontend/app/(main)/explore/chat/[phase]/page.tsx`
  - `src/frontend/lib/i18n/locales/zh.ts`
- 可能修改位置（提示词）：
  - `uidesign/prompt/new-rumination-3.md`
  - `uidesign/prompt/new-rumination2.md`

### P-05 Rumination 最后一步文案“点击1-3行”改为“在以下xxx中选择1-3个”
- 问题分析：
  - 当前步骤7开场引导中已有“点击整行点选1-3个方向”，措辞与目标文案不完全一致。
- 修改计划：
  - 统一步骤7 opening 文案与 i18n 文案，改为“在以下方向中选择1-3个”。
  - 保留“可点击整行”的交互说明作为次句，避免用户不知道如何操作。
- 调整风险：
  - 低风险（文案级）。
- 可能修改位置（代码）：
  - `src/backend/app/domain/rumination_prompt_strings.py`
  - `src/frontend/lib/i18n/locales/zh.ts`
- 可能修改位置（提示词）：
  - `uidesign/prompt/new-rumination2.md`

### P-06 Rumination 匹配阶段：去掉“匹配原因列”，并增强右侧判断标准引导
- 问题分析：
  - 现有后端列定义仍包含`匹配原因`，与需求“仅匹配/不匹配”冲突。
  - 右侧AI判断标准引导不够明确，导致用户难判断。
- 修改计划：
  - 数据结构：步骤2列定义移除`匹配原因`可编辑/展示；保留内部判断时可用隐藏字段（如需要）。
  - 提示词：步骤2 opening 加入“如何判断匹配/不匹配”的标准话术。
  - 前端：表格渲染确认移除后不会影响后续步骤3生成假设输入。
- 调整风险：
  - 中风险：会影响`rumination_ops`与假设生成输入字段，需验证兼容。
- 可能修改位置（代码）：
  - `src/backend/app/utils/rumination_table_widgets.py`
  - `src/backend/app/utils/rumination_ops.py`
  - `src/backend/app/domain/rumination_prompt_strings.py`
  - `src/frontend/components/explore/RuminationTableWidget.tsx`
- 可能修改位置（提示词）：
  - `uidesign/prompt/new-rumination2.md`
  - `uidesign/prompt/new-rumination-3.md`

### P-07 Rumination 各子步骤“深入聊聊”生成内容未遵循提示词
- 问题分析：
  - 当前闸门注入策略以通用 tail 文案为主，步骤差异化不足，容易出现“泛泛而谈”。
- 修改计划：
  - 将步骤2/3/5/6分别拆分为独立 system 片段，并加入字段上下文模板。
  - 对“每条处理完成后引导下一条”的流程加显式约束，防止一次性总结跳步。
  - 增加回归场景（每步至少1条/多条/0条）。
- 调整风险：
  - 中风险：提示词重构后输出风格变化明显，需灰度验证。
- 可能修改位置（代码）：
  - `src/backend/app/utils/rumination_neg_gate.py`
  - `src/backend/app/domain/rumination_step_guidance.py`
  - `src/backend/app/domain/prompts/templates/simple_chat_system.yaml`
- 可能修改位置（提示词）：
  - `uidesign/prompt/new-rumination-3.md`

### P-08 开场引导语需要“冷启动心理预期”
- 问题分析：
  - rumination 开场有目标说明，但“冷启动、可逐步来、不知道也可继续”的心理预期表达仍不稳定。
- 修改计划：
  - 统一开场模板加入冷启动容错句式，并限制长度，避免过度铺垫。
  - 对比前四阶段开场文案口径，统一“允许不确定”的话术风格。
- 调整风险：
  - 低风险（提示词/文案）。
- 可能修改位置（代码）：
  - `src/backend/app/domain/rumination_prompt_strings.py`
  - `src/backend/app/domain/prompts/templates/step_copy.yaml`
- 可能修改位置（提示词）：

## 二、系统代码逻辑问题

### S-01 完成并继续偶发“回到本轮，再点一次才前进”
- 问题分析：
  - 疑似前端状态更新与路由跳转竞态（`stepLocked`/`reportSelectedThreadId`/session保存时机）。
  - 也可能是后端 selected_session_id 未及时写入，前端首跳校验失败。
- 修改计划：
  - 增加“完成并继续”链路日志（前端点击->接口返回->session写入->router.push）。
  - 统一以接口返回为准后再跳转，减少本地推断。
  - 增加去重跳转保护（同 phase 不重复 push）。
- 调整风险：
  - 中风险：牵涉阶段锁定逻辑，改动不当可能放开不该编辑的阶段。

### S-02 Rumination 第一步“优势标记”改为只展示不修改
- 问题分析：
  - 当前步骤1列`优势标记`可编辑，和需求冲突。
- 修改计划：
  - 步骤1改为只读（移除editableCols对应项）。
  - 保留`优势标记`列展示，不隐藏该列。
  - 前端确认按钮可直接用，不做必填校验。
- 调整风险：
  - 低风险：主要是表格列可编辑状态调整。

### S-03 Rumination 热爱表格与已确认关键词不符（合并/拆分/编造）
- 问题分析：
  - 当前关键词提取存在从全文按标点拆分的回退逻辑，可能把推理文本切碎后误当关键词。
  - `extract_dimension_lists_for_rumination_table`多级回退链路中“低置信度来源”未被标记。
- 修改计划：
  - 优先级强制：已确认结论卡关键词 > dimension store > report anchor，降低全文摘词权重。
  - 引入“来源标记+日志”便于定位本次使用了哪一层数据。
  - 对拆分逻辑加白名单规则，避免句子级片段进入关键词。
- 调整风险：
  - 高风险：关系到 rumination 全表输入质量，建议先加观测再替换策略。

### S-04 清缓存后进入已完成阶段仍要重填问卷
- 问题分析：
  - 现有 survey 完成判定混合了本地`session.surveyCompleted`和`surveyApi.getForActivation`兜底，可能受激活码维度影响。
  - 需求是 basic_info 跟 user 绑定，不随激活码变化。
- 修改计划：
  - 登录后先按 user 维度拉取 basic_info 状态，再决定是否跳问卷。
  - 激活页仅保留“无用户basic_info时才去问卷”的判断。
- 调整风险：
  - 中风险：涉及激活与问卷入口统一，需回归多账号切换。

### S-05 清缓存后 journey 不自动定位最新 phase
- 问题分析：
  - 已有`/simple-auth/journeys`与`explore_resume`，但前端可能仍受本地 session 初始值影响。
- 修改计划：
  - 聊天页初始化阶段强制以`explore_resume`覆盖本地阶段，再做路由准入。
  - journey 点击目标 phase 时，优先跳用户点击 phase，校验失败再降级到resume_phase。
- 调整风险：
  - 中风险：需避免用户跳到未解锁阶段。

### S-06 回退到上一阶段后只能重填，且右侧对话未回退
- 问题分析：
  - 目前“回退浏览”和“重新填写”耦合，且消息切片边界与表格step不同步时会出现对话不一致。
- 修改计划：
  - 拆分两条路径：`回看模式(只读)+继续` 与 `重新填写(重置快照)`.
  - 回退动作必须同步切消息边界（step-boundaries）并刷新右侧上下文。
- 调整风险：
  - 高风险：同时影响表格快照、消息裁剪、子步导航状态。

### S-07 “深入聊聊”弹窗重复触发，要求仅首次提交触发
- 问题分析：
  - 现有闸门状态依赖progress中的`pending_table_submit/rumination_neg_state`，重提交流程可能再次触发。
- 修改计划：
  - 每子步增加“首次触发标记”（已触发则仅在“重新填写”后重置）。
  - 明确重置条件：仅`onRefill`或显式重开本步时清除。
- 调整风险：
  - 中风险：需避免误造成“永不触发”。

### S-08 五轮结束后回看第五轮无内容，应默认展示最后阶段结果
- 问题分析：
  - 终态`main_section`进入`recommend/end`后，表格可能被清空或未自动回填step7快照。
- 修改计划：
  - 回看rumination时默认加载step7 submitted快照作为只读结果表。
  - 若无step7，回退到max_reached step。
- 调整风险：
  - 中风险：涉及终态展示与报告页联动。

### S-09 第五轮“工作目的/价值观”来源错乱
- 问题分析：
  - 工作目的下拉来源可能混用了prior文本摘词与dimension_conclusions，导致关键词漂移。
- 修改计划：
  - 固定来源优先级并记录source字段。
  - 前端下拉与右侧对话统一引用同一来源快照。
- 调整风险：
  - 高风险：若历史数据缺失，需定义降级策略。

### S-10 前4阶段“已完成提醒”勾选“不再提醒”后本激活码内永久生效
- 问题分析：
  - 现有提示弹窗存在“本次页面停留不再弹”逻辑，但跨会话持久化需确认粒度。
- 修改计划：
  - 增加“按激活码+phase”的本地持久化键，进入该阶段先读取。
  - 保留管理员调试开关可忽略该状态。
- 调整风险：
  - 低风险。

## 三、UI 设计问题

### U-01 Rumination 表头透明导致内容穿透
- 问题分析：
  - 毛玻璃/透明度叠加导致表头与内容层视觉干扰。
- 修改计划：
  - 提升表头背景不透明度，分离z-index与backdrop效果层。
- 调整风险：
  - 低风险（纯样式）。

### U-02 假设气泡显示不全、滚动条不可用、底部裁切
- 问题分析：
  - 悬浮预览定位与容器边界处理不足，长内容未做可滚动容器控制。
- 修改计划：
  - 改为viewport自适应定位（上/下翻转）。
  - 预览层固定max-height并启用内部滚动。
- 调整风险：
  - 中风险：涉及portal定位与滚动监听。

### U-03 选中假设列宽太窄，三字就换行
- 问题分析：
  - 现有列宽上限偏保守。
- 修改计划：
  - 放宽假设列最小/最大宽度策略，目标支持约15字后换行。
  - 超宽时保留横向滚动。
- 调整风险：
  - 低风险。

### U-04 匹配阶段删除“匹配原因列”
- 问题分析：
  - UI与需求不一致（同P-06）。
- 修改计划：
  - 删除列展示并重排剩余列宽。
- 调整风险：
  - 中风险（与后端字段联动）。

### U-05 Journey 悬停圆圈变大导致被遮挡，整体要更精致
- 问题分析：
  - 当前 hover scale + 固定节点尺寸导致上缘裁切。
- 修改计划：
  - 降低hover放大倍率或取消缩放，改为阴影/描边反馈。
  - 全局缩小node尺寸与间距。
- 调整风险：
  - 低风险。

### U-06 下拉项命名统一（其他->自定义，暂未选定->无，placeholder灰显）及右侧文案同步
- 问题分析：
  - 表格与对话文案存在命名不一致，用户感知割裂。
- 修改计划：
  - 统一文案映射层（前端 i18n + 后端提示词）。
  - placeholder作为不可选项显式灰显。
- 调整风险：
  - 中风险：需兼容历史值映射。

### U-07 热爱列名称左侧多半个括号
- 问题分析：
  - 可能是字段清洗或样式截断造成。
- 修改计划：
  - 排查渲染前字符串清洗与CSS文本截断规则。
- 调整风险：
  - 低风险。

### U-08 点赞后需明显高亮、报告展示点赞内容并落库
- 问题分析：
  - 当前点赞入口存在，但状态反馈和持久化链路不完整。
- 修改计划：
  - 前端：点赞状态改为彩色填充。
  - 后端：点赞只记录`message_id + user_id`，并提供按 message 回查原文能力。
  - 报告页：有点赞内容才展示模块，且展示内容必须可溯源到点赞时的原消息文本。
- 调整风险：
  - 中风险：若消息被删除/重写，溯源可能失败，需要定义“快照留存或容灾兜底”策略。

## 四、其他待看问题

### O-01 最后一轮引导语调用不正确
- 问题分析：
  - 可能出现开场模板来源混用（固定文案/LLM opening/step_copy）。
- 修改计划：
  - 梳理 rumination opening 入口优先级并单测覆盖。
- 调整风险：
  - 中风险：可能影响所有用户首屏体验。

### O-02 重新登录后默认跳转目标（当前是统一“开始探索”）
- 问题分析：
  - 该项已拍板：真实目标是统一跳转到“开始探索”。
- 修改计划：
  - 前后端入口逻辑统一为“登录后进入开始探索页”，移除“自动回到上次页面”的残留逻辑和文案。
- 调整风险：
  - 低风险（实现简单，规则清晰）。

### O-03 激活码归属异常迁移（账号与数据被转移）
- 问题分析：
  - 高风险问题，涉及归属校验与claim_owner流程。
- 修改计划：
  - 排查`is_owner/claim_owner`触发条件与日志，确认是否存在覆盖绑定窗口。
  - 增加关键操作审计日志（code、old_owner、new_owner、来源IP/时间）。
- 调整风险：
  - 高风险（数据安全与账号归属）。

### O-04 登录后个人中心看不到问卷，输入激活码后才显示
- 问题分析：
  - 个人中心可能依赖“当前激活码上下文”而非“用户级basic_info”。
- 修改计划：
  - 个人中心读取接口改为用户级数据直读，激活码仅作为补充来源。
- 调整风险：
  - 中风险：涉及旧数据迁移兼容。

### O-05 多次进入自动新增对话、预览偶发不显示
- 问题分析：
  - 可能由前端初始化时序与本地缓存判断导致“误判无会话即创建”。
- 修改计划：
  - 调整为“后端会话列表确认空后才允许自动新建”。
  - 增加线程列表加载态，避免早渲染空列表。
- 调整风险：
  - 中风险：影响对话入口稳定性。

### O-06 跨设备/清缓存后 journey 消失、需重走流程、删除对话刷新又出现
- 问题分析：
  - 典型“本地缓存与后端记录不同步”问题：删除可能只删本地未删后端，或反之。
- 修改计划：
  - 统一“后端为准”的会话/journey恢复策略；本地仅做加速缓存。
  - 删除对话改为后端成功后再本地删除，并加失败回滚。
- 调整风险：
  - 高风险：涉及数据一致性与用户信任。

## 五、已拍板结论（2026-04-27）

1. 使命阶段可出现多个价值观：**不做结构化多值字段**，只允许用户自然语言填写；**不要求主次**。  
2. 登录后跳转目标：**统一跳到“开始探索”**。  
3. Rumination 第一步“优势标记”：**只展示不修改，且保留该列展示**。  
4. 点赞数据：**仅记录`message_id + user_id`**，但必须保证报告能**溯源到点赞原文内容**。  
5. 使命统计：允许用于**模型内部推理**，但**不展示给用户**。
