# 结论卡状态机梳理与优化建议（v1）

## 1. 背景与结论先行

你现在的核心判断是对的：当前结论卡链路存在“功能叠加过多”的问题，尤其是【确认稿】路径与主对话自动 `STATE_JSON` 路径并行，导致：

- 用户理解成本高（何时等卡、何时点确认稿、何时继续聊）
- 后端状态分支多（`none/pending/rejected/confirmed` + 手动请求确认稿 + 自动 pending）
- 前端交互阻断复杂（末条结论卡未表态阻塞输入、折叠后再引导确认稿）
- ID 语义混用（`report_id/session_id/thread_id`）增加排障难度
- 提示词稳定性受影响（系统注入内容动态变化，cache 命中波动）

建议方向：**取消“确认稿”独立产品入口，收敛为“单一自动结论卡状态机”**，只保留“自动检测可出卡 + 用户确认/再聊聊”两段式体验。

---

## 2. 现状状态机（基于当前代码）

## 2.1 后端结论状态（metadata）

后端目前统一字段（兼容旧字段）：

- `conclusion_state`: `none | pending | rejected | confirmed`
- `conclusion_draft`: 待确认草案（pending 卡）
- `conclusion_final`: 最终结论
- `conclusion_feedback`: 否定反馈
- `thread_completed`: 线程是否完成（部分路径与 confirmed 语义存在耦合但不完全等同）

关键处理点在 `simple_chat_routes.py`：

- 主对话 SSE 中解析 `[STATE_JSON]`
  - `pending_ready` -> 写入 `conclusion_state=pending` + 推送 `dimension_conclusion`
- 若已有 pending，下一轮用户输入先走“判定器”
  - `confirmed` -> 生成最终结论并推送卡
  - `rejected` -> 清 pending 进入 rejected 语义
  - `continue` -> 保持 pending，继续主对话
- `/thread/complete` 最终确认收口并落 anchor/prior context

## 2.2 当前真实状态转换

1. `none/rejected` + 主模型输出 `pending_ready` -> `pending`
2. `pending` + 用户新输入 -> 判定器：
   - `confirmed` -> `confirmed`（并追加结论卡消息）
   - `rejected` -> `rejected`
   - `continue` -> 仍 `pending`
3. 前端点击“确认没有问题” -> `/thread/complete` -> `confirmed + thread_completed=true`
4. 前端点击“我想再聊聊” -> 折叠卡片 + 后端状态置 `rejected` 或 `none`

## 2.3 “确认稿”路径（复杂度来源）

现在还存在独立接口：

- `/conclusion-draft/request`
- `/conclusion-draft/request-stream`

它们会直接生成 pending 草案并落库（不依赖主模型 `pending_ready`），与自动路径并行，造成“双入口同结果”。

## 3. 哪些流程是必须的，哪些可优化

## 3.1 必须保留（不要删）

- **两阶段体验**
  - 阶段 A：正常问答
  - 阶段 B：待确认卡（用户确认或否定）
- **独立判定器**
  - 用户针对 pending 的回复是否“明确确认”，必须和主对话模型职责分离
- **否定后继续聊**
  - `pending -> rejected -> 回到主对话` 是必要闭环
- **最终确认后落库与锚点写入**
  - 影响后续阶段 prior context 与 report 一致性

## 3.2 建议优化（优先级高）

- **移除“确认稿”用户可见入口**
  - 产品层面直接取消按钮与文案
  - 后端保留接口可短期兼容，但不在前端触发（后续可下线）
- **简化状态机到 3 态主干**
  - `collecting`（none/rejected 合并语义）-> `pending` -> `confirmed`
  - `rejected` 作为内部诊断字段，不作为用户心智状态
- **pending 期间弱化提示词注入**
  - 当前 `build_pending_main_dialogue_system_addon` 会动态拼接草案摘要，容易扰动缓存命中
  - 改为固定短标记 + 草案 ID（不注入长文本）
- **减少“阻塞式交互”**
  - 只阻塞“同一张 pending 卡未表态时继续发送”
  - 不再引导“点确认稿再出卡”

## 4. 推荐的新状态机（产品与工程统一）

## 4.1 目标原则

- 对用户：只需理解两件事
  - “AI 还在聊”
  - “AI 给了待确认结论卡，请确认或继续聊”
- 对系统：只保留**单入口出卡**（主模型 `pending_ready`）

## 4.2 简化状态定义

- `collecting`：持续问答，未出现待确认卡（内部可映射 `none/rejected`）
- `pending`：已有待确认卡，等待用户表态
- `confirmed`：最终确认完成，阶段收口

## 4.3 事件驱动

- `collecting --(主模型pending_ready)--> pending`
- `pending --(判定器=confirmed)--> confirmed`
- `pending --(判定器=rejected|continue)--> collecting`
  - 注意：这里建议 `continue` 也回 collecting，而不是长期停留 pending，避免“卡片悬挂”

## 4.4 为什么用户体感更好

- 没有“确认稿”这个新概念
- 不会出现“你先聊、再点确认稿、再确认卡”的三段跳
- 否定后自然回到对话，达到目标再自动出下一张卡

## 5. ID 混用风险与统一建议

你担心的点是成立的，当前确实有多 ID 并存：

- `report_id`: 报告目录主键（`reports/{report_id}`）
- `thread_id`（逻辑会话 ID）: 业务线程，通常是 `t_xxx`
- `session_id`：在不同上下文里有两种含义
  - 激活码记录里的 `rec.session_id`（历史/存储命名空间）
  - 某些消息字段里又被当作 thread id

当前代码已有防护（例如 `resolve_default_logical_thread_id`），但命名仍容易误解。

## 5.1 建议命名标准（强烈建议）

- `report_id`：只表示报告
- `thread_id`：只表示阶段内对话线程
- `activation_session_id`：只表示 activation 存储会话（替代含糊 `session_id`）
- API 入参/出参逐步去歧义（先兼容旧字段，再迁移）

## 5.2 迁移方式（低风险）

1. 后端新增别名字段，不删旧字段
2. 日志统一打印 `report_id/thread_id/activation_session_id`
3. 前端先读新字段，读不到回退旧字段
4. 稳定一个版本后再删旧字段

## 6. 提示词缓存命中风险与改进

你的判断也正确：命中率下降常见原因就是 system prompt 高动态化。

当前动态来源：

- pending 时注入草案摘要/关键词
- rejected 后注入反馈摘录
- 多分支状态提示拼接

## 6.1 改进建议

- **系统提示固定模板化**
  - 固定“状态标签”而非长文本摘要
- **对话事实下沉到普通历史消息**
  - 比 system 追加更稳定
- **限制注入长度与变化频率**
  - 每轮不重复注入同样块；必要时仅注入 hash/短标识
- **观测 cache hit 指标**
  - 已有 `prompt_cache_hit_tokens/miss_tokens` 日志，建议在阶段维度做看板对比改造前后

## 7. 建议的落地改造顺序（两周可控）

## 阶段 1（先降复杂度，不改核心判定）

- 前端下线“确认稿”按钮与引导文案
- 保留后端确认稿接口但不调用（兼容）
- 交互统一为：自动出卡 -> 用户确认/再聊聊

## 阶段 2（收敛状态机）

- `continue` 不再长期维持 pending，回 collecting
- `rejected` 作为内部审计状态，不暴露给用户心智
- 简化 pending 注入内容（固定短模板）

## 阶段 3（ID 统一与可观测性）

- 明确三类 ID 命名
- 关键日志统一打印三元组
- 增加状态迁移埋点（进入 pending、pending 决策结果、最终确认时延）

## 8. 回答你的关键担心（结论）

- **会不会 session_id 错乱？**
  - 有风险，但当前 `ReportRegistry + resolve_default_logical_thread_id` 已有防护；主要问题是“命名歧义”而非“完全无保护”。
- **会不会卡片流程不断新建、导致线程混乱？**
  - 在“确认稿 + 自动 pending”双入口并存时风险更高；收敛为单入口后会显著降低。
- **会不会影响大模型缓存命中？**
  - 会。当前动态注入较多。按上面的模板化/降动态策略可明显改善。
- **是否应该取消确认稿？**
  - **应该。** 从用户体感、工程稳定性、状态可控性三个维度都成立。

---

## 9. 建议的最终产品规则（一句话版）

**用户只看到一个逻辑：正常聊 -> 系统自动在时机成熟时弹待确认结论卡 -> 用户确认或继续聊；不存在“手动生成确认稿”的额外概念。**
