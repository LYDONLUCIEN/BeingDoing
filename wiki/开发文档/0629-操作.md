## 四、数据库迁移 ✅(在开发环境已做)

**操作**:开发环境的 SQLite,从 `003_add_email_verified` 升级到 `005_fix_timestamps_tz`。

### 4.1 备份数据库 ✅

```bash
# 704K,保留作为救命备份
```

### 4.2 停后端 ✅
```bash
./start.sh stop backend
```

### 4.3 修复历史 multiple heads 问题 ✅
**发现的问题**:`002_enhance_likes_table.py` 和 `002_enhance_likes_table_sqlite.py` 用了同一个 revision id(`002_enhance_likes`),导致 alembic 识别出 2 个 head,`upgrade head` 失败。
**修复**:把 sqlite 版移到 `alembic/versions/_archive_duplicate/`(保留不删,alembic 只扫顶层)。已 commit 在 `dccf7e8`。
**说明**:这是历史遗留问题,非本次任务引入。

### 4.4 跑迁移 ✅
```bash
cd src/backend && conda run -n py312 alembic upgrade head
# 003 → 004 → 005 全部执行成功
```

### 4.5 修复 005 version 未记录问题 ✅
**发现的问题**:005 的 UPDATE SQL 执行了(时间数据已修正),但 `alembic_version` 表仍停在 004。
**修复**:`alembic stamp 005_fix_timestamps_tz`(标记版本,数据本身已正确)。

### 4.6 验证迁移结果 ✅
```bash
conda run -n py312 python -c "
import sqlite3
conn = sqlite3.connect('app.db')
print('version:', conn.execute('SELECT version_num FROM alembic_version').fetchone()[0])
print('notification 表:', [t[0] for t in conn.execute(\"SELECT name FROM sqlite_master WHERE type='table' AND name LIKE 'notification%'\")])
print('时间样本:', conn.execute('SELECT created_at FROM users LIMIT 1').fetchone()[0])
"
```
**输出确认**:
- version = `005_fix_timestamps_tz` ✅
- notification 表 = `['notification_tasks', 'notification_recipients']` ✅
- 时间样本 = `2026-02-04 15:28:48.170141+00:00`(带 +00:00)✅
