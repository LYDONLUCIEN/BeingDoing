# data/simple 数据日志说明（ID 去歧义版）

本文档说明 `data/simple` 目录下各类文件的职责、关键字段以及 ID 语义约束，目标是避免 `session_id` 一词多义。

## 1. ID 语义约定（统一口径）

- `report_id`：报告主键，对应目录 `data/simple/reports/{report_id}`。
- `thread_id`：某阶段下的一条对话线程（通常是 `t_xxx`，历史数据也可能是 UUID）。
- `activation_session_id`：激活码体系下的存储会话 ID（历史字段名常为 `session_id`）。

兼容策略：

- 旧字段 `session_id` 暂不删除；
- 新增字段时优先写 `activation_session_id` 或 `thread_id`；
- 读取时优先新字段，缺失时回退旧字段。

## 2. 顶层目录

`data/simple/` 当前主要包含：

- `activations.json`：激活码索引与归属信息（全局入口）。
- `reports/`：按 `report_id` 划分的报告与阶段数据。

## 3. activations.json

路径：`data/simple/activations.json`

结构：`{ activation_code: ActivationRecord }`

常见字段：

- `code`：激活码。
- `session_id`：历史命名，语义应视为 `activation_session_id`。
- `mode`：激活模式（values/strengths/interests/combined）。
- `status`：active/expired/revoked/deleted。
- `owner_user_id` / `owner_email`：归属用户。
- `created_at` / `expires_at` / `last_activity_at`：时间戳。
- `vip_level`、`is_sandbox`、`workspace_*`：沙箱与运行环境信息。

设计理由：

- 激活码是“旅程入口”，需要可追踪归属、有效期、最近活跃时间；
- `activation_session_id`（旧名 `session_id`）用于激活体系内部对齐，不等于某阶段线程 ID。

## 4. reports 目录

路径：`data/simple/reports/{report_id}/`

每个 `report_id` 目录描述一次完整探索旅程（多阶段、多线程）。

### 4.1 record.json（报告主索引）

路径：`data/simple/reports/{report_id}/record.json`

关键字段：

- `report_id`：报告主键。
- `activation_code`：来源激活码。
- `user_id`：当前报告所有者。
- `status`：报告状态。
- `steps.{phase}`：阶段索引（values/strengths/interests/purpose/rumination）。
  - `selected_session_id`：当前阶段选定线程 ID（语义是 `thread_id`）。
  - `session_ids`：该阶段历史线程 ID 列表（语义是 `thread_id[]`）。
  - `locked`：是否已锁定阶段结果。
  - `anchor_summary`：阶段锚点摘要（用于跨阶段上下文）。

设计理由：

- `record.json` 只负责“索引 + 状态”，不保存完整聊天正文；
- 阶段线程与最终选定线程分离，满足审计与回溯。

### 4.2 阶段会话文件（聊天主日志）

路径示例：

- `values__{thread_id}.json`
- `strengths__{thread_id}.json`
- `interests__{thread_id}.json`
- `purpose__{thread_id}.json`
- `rumination__{thread_id}.json`

关键字段：

- 顶层 `session_id`：历史命名，实际是 `report_id`（文件归属）；后续建议新增 `report_id` 直写。
- 顶层 `category`：`{phase}__{thread_id}`。
- `messages[]`：消息序列。
  - `role`：user/assistant/conclusion_card。
  - `event`：user_message/assistant_reply/dimension_conclusion 等。
  - `session_id`：历史命名，消息维度实际语义是 `thread_id`。
  - `step_id`：阶段。
  - `token_usage`：LLM token 统计。
- `metadata`：阶段对话元数据（如结论状态、结论卡草案、最终结论等）。

设计理由：

- 主日志按“阶段 + 线程”拆分，避免单文件无限增长；
- `messages` 中保留 token_usage 便于成本与性能排查；
- `metadata` 负责状态机，不污染主消息内容。

### 4.3 note 事件日志（审计补充）

路径示例：

- `{phase}__{thread_id}__note/note.json`
- 如：`values__{thread_id}__note/note.json`

关键字段：

- `notes[]`：事件级补充日志。
  - `type`：例如 `pending_conclusion_created`、`dimension_conclusion_confirmed`。
  - `payload.thread_id`：关联线程。
  - `created_at`：事件时间。

设计理由：

- 主消息流难以表达“状态迁移原因”，note 负责审计链路；
- 对排查结论卡重复触发、确认路径异常非常关键。

### 4.4 rumination_progress.json（沉淀阶段进度）

路径：`data/simple/reports/{report_id}/rumination_progress.json`

关键字段：

- `schema_version`：结构版本。
- `main_section` / `review_sub_index` / `filter_step`：沉淀步骤定位。
- `filter_table` / `filter_step_snapshots`：表格筛选与每步快照。
- `pending_table_submit`：待提交快照（若有）。

设计理由：

- 沉淀阶段是“结构化筛选 + 多轮修订”，需独立进度文件支持断点恢复；
- 快照可用于回溯用户每一步选择差异。

### 4.5 .locks 目录

路径：`data/simple/reports/.locks/`

用途：

- 文件锁（并发写入保护）目录，防止同一报告被多请求并发改写。

## 5. 当前混淆点与改造建议

当前历史数据中 `session_id` 可能出现三种语义：

- activation 语义（应改称 `activation_session_id`）；
- thread 语义（应改称 `thread_id`）；
- 顶层容器归属（部分文件里实际上是 `report_id`）。

建议按以下节奏推进：

1. 写入路径：新增 `thread_id`、`activation_session_id`，保留 `session_id` 兼容。
2. 读取路径：优先新字段，缺失回退旧字段。
3. 文档/日志：所有接口与日志明确打印 `report_id/thread_id/activation_session_id` 三元组。
4. 稳定版本后：迁移脚本回填并逐步下线歧义 `session_id`。

---

如需严格落地“去除歧义字段”，可再新增一次性迁移脚本，对 `data/simple/reports/*` 做离线回填与字段标准化。
