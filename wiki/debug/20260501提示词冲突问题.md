## A 级（高概率直接造成输出方向冲突）

### 1) 「避免泛化抒情/结构化要点」 vs 「温暖、有温度、自然段收束」 [done]

- 位置：
  - `conclusion_card_goals.py` `get_goal_prompt_hint()`：`请避免泛化抒情，优先输出可核对的结构化要点。`
  - `dimension_completion_checker.py` `_CONCLUSION_GENERATION_SYSTEM`：`summary ... 简短、有温度`
  - `dimension_completion.py` 各 phase `summary_style_rules`：强调 `2-4段自然段`、`温和`、`有陪伴感`
  - `conclusion_card_payload.py` schema 文案：`summary ... 温暖、对用户说话`
- 冲突说明：
  - 同一条 `summary` 同时被要求“结构化要点优先”和“自然温暖叙述”，模型会在“条目化”与“咨询式收束”之间摇摆，风格不稳定。
- 影响：
  - 同一会话内不同轮次 summary 风格漂移（有时像清单，有时像叙事）。
  - 可能降低评测稳定性（尤其对固定格式断言）。
- 建议：
  - 统一口径：要么明确“自然段为主，结构化仅体现在内容可核对”；要么改为“先结构化后一句温和收束”。


### 2) 「keywords 严禁同义替换」 vs 「原词不足可同义词改写」 [done]

- 位置：
  - `dimension_completion_checker.py` `anti_fabrication`：
    - `keywords 必须且只能从对话中用户明确说出的词汇提取`
    - `严禁 ... 同义替换`
  - 同文件 `values_extra`：
    - `实在没有足够原词时，再进行同义词改写`
- 冲突说明：
  - 对同一字段 `keywords` 给出互斥指令（禁止同义改写 vs 允许同义改写兜底）。
- 影响：
  - values 阶段关键词可能被改写，引发“和用户确认词不一致”的回归失败。
- 建议：
  - 二选一：若坚持“只用确认词”，删除 `values_extra` 同义改写句；若允许改写，则同步放宽 `anti_fabrication` 的绝对禁令并改成“仅在用户明确授权时改写”。


### 3) purpose 阶段关键词定义互相打架（10 条经历 vs 1~8 个使命词）[done]

- 位置：
  - `conclusion_card_goals.py` `CONCLUSION_RULES["purpose"]`：`必须是用户为他人提供价值的10个行为或者经历`
  - 同文件 `CONCLUSION_CARD_GOALS["purpose"].validation`：`min_keywords=1`, `max_keywords=8`
  - `conclusion_card_payload.py` purpose schema：`keywords: 1~8 个使命相关核心词或短语`
  - `dimension_completion.py` purpose `goal`：强调“一句话或短语表达使命感”
- 冲突说明：
  - `keywords` 在不同提示中被定义成两类对象：
    - A: 10 条行为/经历
    - B: 1~8 个使命核心词/短语
- 影响：
  - purpose 产出的 `keywords` 语义不确定，前端展示与后续处理容易错位。
  - 解析/校验规则会互相拉扯（数量和内容都不一致）。
- 建议：
  - 明确字段职责：
    - `keywords` 固定为“使命核心词（1~8）”
    - “10条经历”仅放 `experience_value_rows`
  - 然后统一改写 `CONCLUSION_RULES["purpose"]`。


### 4) purpose summary 允许 Markdown 表格 vs 全局禁止提纲式/分条评析 [done]

- 位置：
  - `conclusion_card_payload.py` purpose schema：`summary 可含经历-价值观表格的 Markdown`
  - `dimension_completion_checker.py` `_CONCLUSION_GENERATION_SYSTEM`：`禁止 ... 提纲式分条评析`
  - 同文件 `tone_tight`：`优先 2-4 段自然段`，避免列表化
- 冲突说明：
  - purpose 阶段一边允许表格，一边全局压制结构化分条输出，模型可能随机采用其一。
- 影响：
  - purpose `summary` 格式不稳定（有时纯段落、有时夹表格）。
- 建议：
  - 给 purpose 单独豁免语句（例如“purpose 可附表格，其他阶段禁止列表/表格”）。


### 5) 证据来源范围冲突：只基于“用户消息” vs 额外注入 `basic_info/prior_context` [done]
[这个不是这么理解吧。_CONCLUSION_GENERATION_SYSTEM 的提示词的意思是根据user input的message来输出json。user input的内容包含了basic_info/prior_context  。这没什么问题啊？]

- 位置：
  - `dimension_completion_checker.py` `_CONCLUSION_GENERATION_SYSTEM`：`只根据用户消息里提供的对话记录` 
  - 同文件 `build_conclusion_generation_messages()`：显式注入 `【用户基础信息】` 与 `【前序阶段结论】`
- 冲突说明：
  - 系统层说“只用用户消息”，用户层又提供了非当前对话正文的额外上下文。
- 影响：
  - 模型对“可引用证据边界”理解不一致，可能导致忽略注入信息或误用外部信息。
- 建议：
  - 把系统句改为“仅基于当前提供的上下文块（对话+基础信息+前序结论）”。

## B 级（中概率造成风格/结构抖动）

### 6) 「避免列表化」 vs strengths 明确要求“3条以内建议” [done]

- 位置：
  - `dimension_completion_checker.py` `tone_tight`：避免“第一点/第二点”式罗列
  - `dimension_completion.py` strengths `summary_style_rules`：`最后给出3条以内建议`
- 冲突说明：
  - 一个要求限制罗列感，一个要求“3条建议”，模型在“段落建议”与“条目建议”间摇摆。
- 建议：
  - 改成“3条建议可写在自然段中，不必编号”并写入统一规则。


### 7) 完成判定与生成文风之间的目标不一致（确认导向 vs 洞察导向）

- 位置：
  - `dimension_completion.py` `completion_criteria`：强调“用户确认/认可即完成”
  - 同文件 `summary_style_rules`：允许“轻度洞察归纳”“风险提醒”等
- 冲突说明：
  - 判定通过条件偏“流程确认”，生成要求偏“咨询洞察”；在信息不足但已确认时，输出容易过度发挥。
- 建议：
  - 增加兜底规则：“若可核对素材不足，优先回放已确认事实 + 1条最小行动建议”。

## C 级（当前可能未激活，但存在配置漂移风险） [done]

### 8) `pending_conclusion_reply.yaml` 协议与主链路 `STATE_JSON` 协议不一致

- 位置：
  - `pending_conclusion_reply.yaml`：要求 `[REPLY] + [CONCLUSION_JSON]`
  - `prompt_builder.py`：主协议为 `[STATE_JSON] {"state":"...","draft":...}`
- 现状判断：
  - 代码中 `get_pending_conclusion_injection()` 目前未见实际调用点（仅 loader/export 存在）。
- 冲突说明：
  - 若后续恢复接入旧模板，会与当前主协议冲突，导致解析失败或状态机异常。
- 建议：
  - 要么删除该模板，要么改成与 `STATE_JSON` 完全一致的格式并补回归测试。

## 建议优先级（修复顺序）

1. **先修 A3（purpose 字段语义冲突）**：这是结构层面的硬冲突，最容易导致数据错位。  
2. **再修 A2（同义替换互斥）**：直接影响关键词真实性与回归稳定性。  
3. **然后修 A1/A4（文风与结构冲突）**：统一 summary 输出范式。  
4. **最后处理 A5/C8**：清理上下文边界和遗留协议，防止后续二次回归。
