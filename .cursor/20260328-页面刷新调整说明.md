# 2026-03-28 页面刷新与登录态调整说明

## 1. 目标与范围

本次调整目标：

1. **用户体验**：支持长期登录（30 天），减少“思考中断后必须重登”的打断感。
2. **安全性**：将长生命周期凭证从前端可读存储迁移到 `HttpOnly Cookie`，降低 XSS 窃取风险。
3. **稳定性**：并发请求场景下通过单飞刷新（single-flight）避免 refresh 风暴。

本次仅涉及“认证与页面刷新体验”相关链路，不改变业务功能逻辑。


## 2. 版本变动摘要（Auth Session Upgrade）

### 2.1 核心策略变化

- Access Token：改为短期（默认 30 分钟）。
- Refresh Token：长期（默认 30 天）。
- Refresh Token 存储：从 `localStorage` 升级为 `HttpOnly Cookie`。
- 401 自动处理：前端自动 refresh 一次，成功后重放原请求，失败再弹登录。
- 页面空闲提示：默认 30 分钟未操作提示刷新，但不直接清登录态。

### 2.2 后端变更

- 配置新增（`src/backend/app/config/settings.py`）：
  - `ACCESS_TOKEN_EXPIRE_MINUTES`
  - `REFRESH_TOKEN_EXPIRE_DAYS`
  - `REFRESH_TOKEN_ROTATE`
  - `REFRESH_TOKEN_SECRET_KEY`
  - `REFRESH_COOKIE_NAME`
  - `REFRESH_COOKIE_DOMAIN`
  - `REFRESH_COOKIE_PATH`
  - `REFRESH_COOKIE_SAMESITE`
  - `REFRESH_COOKIE_SECURE`

- 新增 refresh token 持久化模型：
  - `src/backend/app/models/refresh_token.py`
  - 已加入 `src/backend/app/models/__init__.py`

- 认证服务增强（`src/backend/app/services/auth_service.py`）：
  - refresh token 创建/持久化/轮换/撤销
  - family 级重放检测基础防护
  - 启动后按需自动创建 `refresh_tokens` 表（`checkfirst=True`）

- 认证路由升级（`src/backend/app/api/v1/auth.py`）：
  - `POST /api/v1/auth/login`：写入 refresh cookie
  - `POST /api/v1/auth/register`：写入 refresh cookie
  - `POST /api/v1/auth/refresh`：优先从 cookie 读取 refresh token
  - `POST /api/v1/auth/logout`：撤销 refresh token 并清除 cookie
  - 兼容灰度：`refresh/logout` 仍兼容 body 传 refresh_token（旧客户端不立即失效）

### 2.3 前端变更

- `src/frontend/lib/api/client.ts`
  - `axios` 启用 `withCredentials: true`
  - 401 自动 refresh（single-flight）+ 重试原请求
  - 移除对 `localStorage.refresh_token` 依赖

- `src/frontend/stores/authStore.ts`
  - 移除 `refreshToken` 状态与持久化，仅保留 access token（短期）

- `src/frontend/lib/api/auth.ts`
  - 登录/注册不再处理 refresh_token 字段
  - refresh/logout 调用改为无 body token，由 cookie 驱动

- `src/frontend/components/layout/AuthModal.tsx`
  - 登录/注册仅写入 access token

- `src/frontend/components/layout/TopNavbar.tsx`
  - logout 调用后端接口撤销 refresh
  - 新增空闲 30 分钟刷新提示（`NEXT_PUBLIC_IDLE_REFRESH_MINUTES` 可配）

## 4. 验收操作清单（手工）

> 建议在“测试环境”先完整走一遍，再进生产。

### 4.1 登录与 Cookie 验证

1. 打开登录页，使用测试账号登录。
2. 在浏览器开发者工具查看 `Application/Cookies`：
   - 存在 refresh cookie（默认名 `bd_refresh_token`）
   - Cookie 带 `HttpOnly`
3. 查看 `Local Storage`：
   - 仅有 `token`
   - 不应再有 `refresh_token`

验收标准：满足以上三点。

### 4.2 Access 过期后自动续签

1. 人工将 `ACCESS_TOKEN_EXPIRE_MINUTES` 设为很短（例如 1 分钟）并重启服务（仅测试环境）。
2. 登录后等待 access 过期，再触发任意受保护 API（如个人主页请求）。
3. 观察网络请求：
   - 先出现一次 401
   - 自动调用 `/api/v1/auth/refresh`
   - 原请求自动重试成功（200）

验收标准：用户无感继续使用，无需手动重登。

### 4.3 refresh 失效后的退出行为

1. 手动清除 refresh cookie 或等待其过期（测试可主动清）。
2. 再触发需要登录的请求。

验收标准：refresh 失败后前端清理登录态并弹登录，不出现无限重试。

### 4.4 并发刷新稳定性

1. 在 access 临近过期时，页面同时触发多个并发请求。
2. 观察网络面板 refresh 调用次数。

验收标准：同一波 401 仅一次 refresh（single-flight），其余请求复用结果。

### 4.5 退出登录一致性

1. 点击退出登录。
2. 校验：
   - 前端 token 清理
   - refresh cookie 被删除
   - 再访问受保护接口触发登录

验收标准：退出后不应仍保持可访问受保护资源。

## 7. 已知风险与监控建议

### 7.1 风险点

- 如果生产误设 `REFRESH_COOKIE_SECURE=False`，安全性下降。
- 跨域部署若 `withCredentials/CORS` 不一致，会出现 refresh 不生效。
- 多实例部署时需保证数据库连接稳定，否则 refresh 持久化会异常。

### 7.2 建议监控

- `/auth/refresh` 成功率、失败率
- 401 频次与重试成功率
- logout 后仍可访问受保护资源的异常日志


## 8. 回滚策略（应急）

若线上出现异常，可临时回滚到“纯 access + 手动重登”模式：

1. 前端关闭自动 refresh 逻辑（回退到原 401 处理）。
2. 后端保持登录接口可用。
3. 保留 refresh 表与字段（不删库），待排查后再恢复。
