# 2026-03-28 Admin 调试空间开发报告

## 1. 背景与目标

本次开发围绕以下核心诉求展开：

1. **管理员调试能力独立于普通用户流程**  
   普通用户仍保持原有约束（激活码、阶段顺序、不可回退、每步会话数限制等），管理员在调试工作区可绕过这些限制。
2. **同一套前后端服务，不分叉业务主链路**  
   管理员调试仍走 `simple-auth/simple-chat` 主链路，保证“调试效果 = 实际效果”。
3. **特权可控、可关闭**  
   生产环境默认不启用特权；通过统一策略开关一键关闭，而不是散落在多个 if 判断里。
4. **提示词在线实验（sandbox_only）**  
   在 Admin 页面进行提示词版本管理与绑定测试，仅对 `SBX/ADM` 调试工作区生效，不影响普通用户。


## 2. 整体设计思路

### 2.1 总体原则

- **统一抽象**：将调试空间统一为 `workspace` 概念，避免 “沙箱是沙箱、常驻是常驻” 的双轨逻辑。
- **最小侵入**：不重构主流程，优先在边界处加策略和扩展字段。
- **兼容优先**：兼容旧数据结构（历史 `SBX` 记录）、已有 Fork 机制、已有前端页面。
- **安全优先**：所有调试特权受 `super_admin + policy 开关 + workspace 类型` 三重约束。

### 2.2 关键抽象

1. **Workspace 类型**
   - `fork`：现有 `SBX` Fork 沙箱（短期、可过期清理）
   - `resident`：新增管理员常驻工作区（长期、可复用）

2. **Policy 层（统一开关）**
   - 总开关：`ADMIN_DEBUG_POLICY_ENABLED`
   - 子开关：
     - `ADMIN_DEBUG_WORKSPACE_ENABLED`
     - `ADMIN_SANDBOX_ENABLED`

3. **Prompt Lab（sandbox_only）**
   - profile（提示词配置集） -> version（版本） -> activation 绑定
   - 运行时只对 `SBX/ADM` 生效

## 4. 关键流程说明

### 4.1 管理员调试工作区

1. Admin 点击“进入我的常驻调试工作区”
2. 后端 `workspace/ensure` 返回 `ADM...`
3. 前端跳转 `/explore/activate?code=ADM...`
4. 激活页识别管理员 + 策略开关 -> 自动解锁并进入 chat
5. chat 侧绕过 step 锁定逻辑

### 4.2 Prompt Lab 生效链路（sandbox_only）

1. Admin 在 Prompt Lab 创建 profile & 版本
2. 绑定 profile 到 `SBX/ADM` 激活码
3. 对话请求进入 `simple_chat`：
   - 满足 sandbox_only 条件才读取绑定并覆盖提示词
4. 普通激活码/普通用户继续使用代码内默认模板

### 4.3 回填到代码模板

1. Prompt Lab 点击“复制纯模板”或“复制模板+goals”
2. 手工粘贴回 `domain/prompts/templates/*.yaml` 对应模板
3. 以代码变更方式纳入版本管理（可审查、可回滚）


## 5. 安全与边界控制

1. **默认关闭特权**：`ADMIN_DEBUG_POLICY_ENABLED=False`
2. **身份控制**：仅 super_admin 可调用调试 API
3. **范围控制**：Prompt Lab 仅 `SBX/ADM` 绑定，普通激活码不可绑定
4. **运行时双重判断**：API 层 + simple-chat 层均有策略判断，防止旁路调用
5. **兼容旧数据**：对历史 `is_sandbox` 记录提供兼容识别
