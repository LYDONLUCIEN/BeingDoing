# Rumination 新版交互 — 设计评估初稿

> 文档日期：2026-03-29  
> 依据：`rumination_prompt.md`、`.cursor/20260329-rumination过滤机制.md`、以及本次产品设想（表格为中心、按列/按步确认、行级对话、折叠列等）。  
> **说明**：本文为评估与决策清单；**不要求与当前仓库实现一致**。

---

## 一、新版核心设想（摘要）

1. **表格为页面主题**：筛选阶段以「一张持续演进的表」为主 UI，对话为辅（或分栏：表 + 聊）。
2. **Step 状态机由「表格确认」驱动**：每步仅允许编辑指定列；底部「确认本步」后进入下一步；已确认步在表头展示 **绿色 √**、样式区分且 **不可再编辑**。
3. **后端表数据与 UI 同步**：任何列内容变更需 **持久化**（粒度见下文决策点）；保证刷新、多设备一致。
4. **行级上下文对话**：用户可选中一行，针对该行做 **问答/反思**；记录进入 **会话历史**（与主对话的关系需定义）。
5. **步骤 3 类交互**：「用户确认的假设」支持 **快速选假设1/2/3** 或选 **其他 → 自由输入**。
6. **体验**：减少上下滚动；过滤后 **中间列默认折叠**，可手动展开查看历史列。

---

## 二、与当前实现的差异（方向性）

| 维度 | 当前实现（概括） | 新版目标 |
|------|------------------|----------|
| 表格入口 | 依赖 `main_section=filter` 等条件，与对话衔接有断层 | 表格常驻或显式「筛选工作台」，状态与步骤强绑定 |
| 编辑粒度 | 曾支持整表或单行 patch | 按 **Step 锁定列** + 可选 **按列确认** 子状态 |
| 行级聊天 | 无独立「选中行 → 上下文」协议 | 需 **row_id / 列版本** 与消息关联 |
| 表头状态 | 无「已确认步」视觉协议 | 每步对应列组绿色 √ + 只读 |
| 列可见性 | 列随步骤变化，前端一次性展示 | **折叠/展开** 历史列，降低视觉负担 |
| 假设选择 | 多为纯文本格 | **下拉：假设1/2/3/其他** + 其他时输入框 |

## 三、需要改动的大类（不涉及具体文件）

1. **状态模型**：`filter_step`（1–9）是否足够；是否要 `sub_step` / `active_column_group`；「按列确认」是否拆子状态。
2. **API**：除现有 submit 外，是否需要 **按单元格/按行自动保存**、**行级聊天专用 endpoint** 或统一在 stream 里带 `row_context`。
3. **前端信息架构**：单页分栏（表+聊）还是 Tab；表格 **虚拟滚动/冻结首列**；折叠列的 **默认展开规则**。
4. **对话存储**：行级消息写入 **同一线程**（带 metadata）还是 **子线程**；历史列表如何展示「针对第 N 行」。
5. **步骤 3 UI**：下拉选项与单元格值的 **单一数据源**（避免显示文案与存储不一致）。

## 四、风险与问题

### 4.1 产品/交互

- **「一步」与「一列」**：md 原文是 **一步 = 一类可编辑列**；若再拆成「每列单独确认」，步数会膨胀，需与用户心智对齐。
- **行级对话与「一次一问」**：主流程仍要求咨询师 **一次一问**；若用户频繁切行聊天，可能 **冲淡主线**，需要 UI 上区分「探索性对话」与「主对话」。
- **折叠列**：折叠后用户可能 **找不到** 自己当初为什么删某行；需保留 **「查看本行历史快照」** 或展开入口说明。

### 4.2 技术

- **并发保存**：快速改多格 + 未确认步若都写库，需 **乐观锁或版本号**，避免覆盖。
- **步骤 9 删行**：若 UI 支持删行，需与后端 **行 id 稳定性**、引用行聊天记录的 **孤儿消息** 策略一致。
- **LLM 假设列**：生成失败、超时时的 **占位与重试** 与「确认本步」的禁用条件。
- **性能**：行数 × 列数 × 折叠渲染；大数据表需 **分页或虚拟列表**。

### 4.3 数据与合规

- 行级聊天可能含 **敏感自我披露**；是否与主对话 **同一保留策略**、导出报告是否包含。

## 五、需要你决策的事项（建议优先级）

1. **确认粒度**  
   - A：仅 **每步一次确认**（与 md 一致，实现简单）  
   - B：**每列一次确认**（更细，状态机更复杂）  
   - C：混合（例如步骤 3 内先锁假设选择再锁别的——需列清单）

2. **行级对话的数据归属**  
   - A：全部进 **当前 rumination 线程**，消息带 `row_id`  
   - B：**独立子会话** 按行挂载，主对话仅嵌入摘要  
   - C：仅 **前端暂存**，确认步前再合并（一般不推荐）

3. **自动保存 vs 仅确认保存**  
   - A：改格即 **PATCH**（体验好，请求多）  
   - B：仅点确认 **批量提交**（与现后端接近，断电风险）  
   - C：防抖自动保存 + 确认时 **校验快照**

4. **过滤后列的默认展示**  
   - A：只显示 **当前步相关列** + 「展开历史列」  
   - B：全列窄表横向滚动（实现简单，体验差）  
   - C：时间轴式「步骤卡片」回顾（开发量大）

5. **步骤 3「其他」**  
   - A：下拉选「其他」后 **同一单元格变输入框**  
   - B：弹层/侧栏编辑，格内只显示摘要  
   - C：选假设1/2/3 直接 **拷贝文案** 到可编辑格，用户可再改

6. **0 行提前结束**  
   - 各过滤后 0 行：是 **阻断确认**、**允许回退到上一步**，还是 **结束筛选并回主对话说明**？

7. **主页面回顾四步**  
   - 仍 **纯对话** 还是也要 **小卡片/清单只读展示 prior**（与表格工作台风格统一）？
